# Fitness & Bioimpedance Tracker — Handoff / Spec

Документ для переноса прототипа в полноценную серверную версию (Claude Code).
Текущий прототип — рабочий React-артефакт (`fitness-tracker.jsx`). Цель — сервер на Beget с БД, авторизацией и синхронизацией между устройствами.

-----

## 1. Что это и текущее состояние

Личный трекер силовых тренировок и состава тела (биоимпеданс). Один пользователь. Ввод весов/повторов, история, графики прогресса.

**Сделано (прототип):** одностраничное React-приложение, 4 вкладки, данные в `window.storage` (встроенное хранилище артефактов Claude). Прототип отвечает на «что и как должно работать».

**Главные ограничения прототипа — то, что чиним на сервере:**

- `window.storage` **не синхронизируется между устройствами** (на телефоне пусто). Не настоящий бэкенд.
- Нет авторизации.
- Нет надёжного бэкапа (риск потери при чистке браузера).
- Историческая выгрузка из Excel вшита прямо в код константой (костыль для разовой миграции).

-----

## 2. Стек прототипа

- React (функциональные компоненты, hooks).
- `recharts` — графики.
- `lucide-react` — иконки.
- Шрифты: Bricolage Grotesque (заголовки), Hanken Grotesk (текст), JetBrains Mono (цифры).
- Тема: тёмная (charcoal `#0c0e0a`) + лаймовый акцент `#c8f23f`. Все CSS-переменные/классы в одном `<style>`-блоке внутри компонента + объект `C` для цветов в recharts.
- Хранилище: обёртка `store.get/set` над `window.storage` (JSON в строках).

## 3. Функциональные требования (по вкладкам)

### Обзор (`Home`)

- Карточки-метрики: тренировок за 7 дней, всего тренировок, текущий вес (+дельта к прошлому замеру), жир % (+дельта).
- Дельты по весу/жиру: снижение = «хорошо» (зелёная стрелка вниз) — инвертированная логика.
- Блок «Последняя тренировка»: тип + дата + список упражнений (топ-вес и объём по каждому).
- Блок «Состав тела»: последний замер (6 метрик).
- Если данных нет — кнопки-заглушки, ведущие на нужную вкладку.

### Тренировки (`Log`)

- Выбор одного из 4 шаблонов (см. §6). Под каждым: дата последней такой тренировки («посл.: 17 мая» / «ещё не было») и бейдж **«следующая»** на той, что идёт по циклу.
- Логика «следующей»: цикл `h1t1 → h1t2 → h2t1 → h2t2 → (повтор)`. Берётся самая свежая сессия, следующая в массиве шаблонов. Нет сессий → первая.
- Поле даты (по умолчанию сегодня).
- Карточки упражнений с **преднастроенными целевыми весами/повторами** из программы (плейсхолдеры подсказывают диапазон, напр. «8–10»).
- Внутри упражнения: добавить/удалить подход; удалить упражнение целиком (если не делал — **важно**, т.к. цели преднастроены и иначе сохранятся плановые значения).
- Сохранение: тост-подтверждение «{тип} сохранена — {дата}». Пустые подходы отфильтровываются. Если всё пусто — подсказка.
- История (раскрывается): список сессий (тип, дата, суммарный объём, упражнения с подходами). Удаление через **инлайн-подтверждение** (двухшаговое), т.к. нативный `confirm()` заблокирован в песочнице — на сервере можно вернуть обычный confirm/модалку.
- Кнопка одноразового импорта вшитой истории (`SEED_HISTORY`) — **на сервере убрать**, заменить сидингом БД.

### Тело (`Body`) — биоимпеданс

- Форма замера: дата + 6 метрик (см. §4). Любое поле опционально.
- Замер на ту же дату **перезаписывается** (один замер в день).
- История замеров с инлайн-удалением.
- Тост-подтверждение сохранения.

### Прогресс (`Progress`)

- Выбор упражнения (dropdown, только встречавшиеся в истории) → график **объёма** или **макс. веса** по сессиям (bar).
- Состав тела: переключатель метрики → линейный график по датам.
- Упражнения матчатся **по точному названию** → отсюда требование единых названий (см. §5).

## 4. Модель данных

```jsonc
// Сессия тренировки
{
  "id": "string",            // уникальный; в проде — UUID/autoincrement
  "date": "YYYY-MM-DD",
  "templateId": "h1t1|h1t2|h2t1|h2t2",
  "exercises": [
    {
      "n": "string",         // КАНОНИЧЕСКОЕ название упражнения
      "sets": [
        { "weight": 70, "reps": 8 }   // weight: число | "" (свой вес) | 0 (свой вес)
        // reps: число | "" | строка ("до отказа")
      ]
    }
  ]
}

// Замер биоимпеданса
{
  "id": "string",
  "date": "YYYY-MM-DD",      // уникальна (upsert по дате)
  "weight": 0,  "fat": 0,  "muscle": 0,
  "water": 0,   "visceral": 0, "bone": 0,
  "note": ""                 // все метрики: число | null
}
```

Биометрики (ключ/подпись/ед.изм.):
`weight`/Вес/кг · `fat`/Жир/% · `muscle`/Мышцы/кг · `water`/Вода/% · `visceral`/Висц. жир/— · `bone`/Кости/кг.

### Бизнес-правила

- **Объём упражнения** = Σ(weight × reps) по подходам (свой вес → 0).
- **Топ-вес** = max(weight) по подходам.
- **Дельта** на дашборде = последний замер − предыдущий.
- Невалидные числа → трактуются как 0/null (через хелпер `num`).

## 5. Канонические названия упражнений (важно!)

Из-за разных формулировок в Excel и в программе возникали дубли. Введён словарь синонимов → канон. На сервере его сохранить (нормализация при сейве + миграция при импорте). Можно вынести в справочник `exercises` в БД.

|Синонимы (входящие)                                                    |Канон                               |
|-----------------------------------------------------------------------|------------------------------------|
|Жим гантелей лёжа; Жим гантелей под углом 30°                          |**Жим гантелей 15°**                |
|Жим лёжа                                                               |**Жим штанги лёжа**                 |
|Жим в тренажёре на грудь (1 рука)                                      |**Жим в тренажёре сидя на грудь**   |
|Махи в стороны                                                         |**Махи гантелей в стороны**         |
|Молотковые                                                             |**Молотковые подъёмы гантелей**     |
|Бицепс штанга                                                          |**Подъём штанги на бицепс**         |
|Отжимания на брусьях                                                   |**Брусья**                          |
|Подтягивания в гравитроне                                              |**Подтягивания**                    |
|Подъёмы на носки в тренажёре сидя                                      |**Подъём на носки в тренажёре сидя**|
|Подъём ног в висе или dead bug                                         |**Подъём ног в висе**               |
|Протяжка к подбородку                                                  |**Протяжка штанги к подбородку**    |
|Разведения лёжа                                                        |**Разведения гантелей лёжа**        |
|Бабочка / сведения; Сведения в тренажёре бабочка; Сведения в кроссовере|**Сведения в тренажёре**            |
|Трицепс в блоке / канат                                                |**Трицепс в канате**                |

`normSession()` дополнительно объединяет упражнения с одинаковым каноном **внутри одной сессии** (склеивает подходы).

**Спорные слияния (потеря нюанса) — на ревью продукта:**

1. «Жим гантелей 15°» теперь включает плоский (лёжа) и 30° наклон.
1. «Сведения в тренажёре» = бабочка + кроссовер (разные тренажёры).
1. «Подтягивания» = свободные + гравитрон (в гравитроне число = вес помощи, смысл нагрузки обратный).

«Жим гантелей сидя» оставлен **отдельно** (это плечи, не грудь).

## 6. Программа тренировок (4 шаблона)

Преднастроенные цели для логирования. Формат подхода: `вес×повторы` (диапазон → в подсказке). Названия ниже даны как в исходной программе — **в коде/БД хранить уже канонизированными** (§5).

**Н1·Т1 (грудь/спина/руки):** Выпрыгивания с гантелью 12×5 ×4 · Жим лёжа 60×8/70×5/70×5(4–5)/70×4 · Разведения лёжа 14×10 ×3 · Подтягивания 6 б/п / 5–6 б/п / помощь 14×8–10 · Махи в стороны 9×12 ×3 · Бицепс штанга 20×10/20×10/17.5×10 · Подъём ног в висе 12×3

**Н1·Т2 (ноги/плечи/трицепс):** Подъёмы на носки сидя 20×15/20×15/25×12 · Жим гантелей 15° 22×10 ×3 · Тяга верт. блока к груди 45×12/45×12/45×10 · Брусья 12 б/п / 8–10 б/п / помощь 9×9–10 · Жим гантелей сидя 16×10 ×3 · Трицепс канат 11.3×12/11.3×12/13.5×8–10 · Пресс в тренажёре 12×3

**Н2·Т1 (грудь/спина/руки):** Гоблет-присед 26×8 ×4 · Жим гантелей 15° 22×10/24×8/24×8 · Бабочка/сведения 15×12/15×10–12 ×2 · Подтягивания 8 б/п / 5 б/п / помощь 14×8–10 · Протяжка к подбородку 25×10 ×3 · Молотковые 10×10/10×10/7.5×10 · Подъём ног в висе / dead bug ×3

**Н2·Т2 (ноги/плечи/руки):** Подъёмы на носки сидя 20×15/25×12/25×12 · Жим в тренажёре на грудь (1 рука) 30×10/30×10/32.5×8–10 · Тяга гориз. блока к поясу 45×10/45×10/50×8–10 · Брусья 10/10/9–10 б/п · Жим гантелей сидя 16×10/18×8/18×8 · Подъём штанги на бицепс 25×10 ×2/25×8–9 · Пресс в тренажёре 12×3

Точные массивы целей — в константе `TEMPLATES` в `fitness-tracker.jsx`.

## 7. Импортированная история (контекст)

10 тренировок (21.03.2026 – 17.05.2026) загружены из `fitness-trains.xlsx`. Нюансы парсинга — учесть при будущих импортах:

- Дата в Excel — serial number (конвертация: `1899-12-30 + N дней`).
- **Excel мангленил десятичные веса в даты** (`11.3` → «11 мар»). Восстановлено правилом `день.месяц`, сверено с программой (Трицепс 11.3/13.5/12.5; жим тренажёр 27.5; брусья 8.5 повт). При новых импортах: колонки веса/повторов в Excel держать в формате **Текст/Число**, иначе повторится.
- `session_id` одной записи содержал «2025-03-27», но колонка даты = 2026-03-27 → доверяем колонке даты.
- Колонка `exercise_order` содержала мусор — игнорируется.
- Колонки исходника: `session_id, date, week, training, exercise_order, exercise, set_no, weight_kg, reps`.
- `week`(1/2) + `training`(1/2) → `templateId` (`h{week}t{training}`).

На сервере историю **засидить в БД миграцией** (из `SEED_HISTORY` в коде или заново распарсив xlsx), а не вшивать в клиент.

## 8. Целевая архитектура (сервер, Beget)

Beget поддерживает: PHP 8.x, MySQL, PostgreSQL, Node.js, phpMyAdmin, бесплатный SSL, ежедневные бэкапы 30 дней.

### Рекомендация

- **Фронтенд:** тот же React, собрать через **Vite** в статику, выложить в корень домена. Логику (шаблоны, канон, цикл, расчёты) перенести как есть.
- **Бэкенд (основной вариант — наименьшее трение на shared-хостинге):** **PHP 8 + MySQL**, API в `/api/` того же домена (без CORS). PDO + prepared statements.
- **Альтернатива:** Node.js + SQLite/MySQL (если возьмёшь VPS и хочешь единый JS-стек).
- **Хранилище данных в JSON-колонке хранить как `TEXT`** (Beget MySQL может быть 5.x без native JSON).

### Замена хранилища

Сейчас весь доступ к данным идёт через обёртку `store` (`store.get/set`). На сервере — заменить её на модуль `api` с `fetch`. Структура состояния и компоненты не меняются.

### API-контракт (REST, бэкенд-агностичный)

```
Авторизация: заголовок  Authorization: Bearer <TOKEN>   (общий секрет в env)

GET    /api/sessions            -> [Session]
POST   /api/sessions            <- Session   (upsert по id)
DELETE /api/sessions/{id}

GET    /api/bio                 -> [BioEntry]
POST   /api/bio                 <- BioEntry   (upsert по date)
DELETE /api/bio/{id}

GET    /api/export              -> { sessions:[...], bio:[...] }   (бэкап)
POST   /api/import              <- { sessions:[...], bio:[...] }   (восстановление/миграция)
```

### Схема БД (MySQL)

```sql
CREATE TABLE sessions (
  id          VARCHAR(40) PRIMARY KEY,
  date        DATE NOT NULL,
  template_id VARCHAR(8) NOT NULL,
  data        TEXT NOT NULL,          -- JSON: exercises[]
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  INDEX (date), INDEX (template_id)
);

CREATE TABLE bio_entries (
  id         VARCHAR(40) PRIMARY KEY,
  date       DATE NOT NULL UNIQUE,    -- один замер в день
  weight     DECIMAL(5,1), fat DECIMAL(4,1), muscle DECIMAL(5,1),
  water      DECIMAL(4,1), visceral DECIMAL(4,1), bone DECIMAL(4,1),
  note       TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- опционально: справочник единых названий
CREATE TABLE exercise_aliases (
  alias  VARCHAR(120) PRIMARY KEY,
  canon  VARCHAR(120) NOT NULL
);
```

### Безопасность (обязательно)

- Без авторизации API на публичном домене = любой читает/пишет твои данные.
- Для одного пользователя достаточно **общего секретного токена** (в env, проверка на сервере) или **Basic Auth через `.htaccess`**.
- HTTPS включить (бесплатный SSL Beget).
- Не коммитить токен/доступы к БД в репозиторий.

### Надёжность / бэкап

- Данные в БД + ежедневные бэкапы Beget (30 дней).
- Эндпоинты `/export` + `/import` как ручной бэкап и инструмент миграции с прототипа.

## 9. План для Claude Code (по шагам)

1. Инициализировать репозиторий, вынести React-компонент из артефакта в Vite-проект (`src/App.jsx`), подключить `recharts`, `lucide-react`, шрифты.
1. Вынести `store` → модуль `api.js` (fetch + токен). Заменить вызовы `store.get/set` на `api.*`.
1. Добавить экран ввода токена (gate) + хранение токена локально.
1. Бэкенд: PHP `/api/*` (или Node), PDO+MySQL, prepared statements, проверка токена в middleware.
1. Миграции БД (схема §8). Сид истории из `SEED_HISTORY`/xlsx.
1. Перенести нормализацию названий (§5): применять при сейве и в `/import`.
1. Сборка Vite → статика в корень домена, API в `/api/`. Включить SSL.
1. Реализовать `/export` `/import`; одноразово перенести ручные данные с прототипа (если нужны).
1. Вернуть нативный `confirm`/модалку для удаления (инлайн-подтверждение было костылём под песочницу).
1. Убрать из клиента вшитую `SEED_HISTORY` и кнопку импорта.
